## PA3-3分页机制

### 实验问题

#### 1.Kernel的虚拟页和物理页的映射关系是什么？请画图说明

Kernel采用线性映射：
- 虚拟地址范围：`0xc0000000~0xc0ffffff`
- 物理地址范围：`0x00000000~0x00ffffff`
- 映射关系：虚拟地址=物理地址+0xc0000000

```
虚拟地址空间                物理地址空间
0xc0ffffff  ┐              0x00ffffff  ┐
            │ Kernel区                  │ Kernel区
            │ (16MB)                    │ (16MB)
0xc0000000  ┘──────────────> 0x00000000  ┘
```
（该图绘制运用ai）

#### 2.以某一个测试用例为例，画图说明用户进程的虚拟页和物理页间映射关系又是怎样的？Kernel映射为哪一段？你可以在loader()中通过Log()输出mm_malloc的结果来查看映射关系，并结合init_mm()中的代码绘出内核映射关系。

用户进程映射关系：
- 虚拟地址：从`0x08048000`开始
- 物理地址：从`0x01000000`开始分配
- Kernel映射：`0xc0000000~ 0xc0ffffff`映射到`0x00000000 ~ 0x00ffffff`

```
虚拟地址空间                物理地址空间
0xffffffff                 0x08000000 (128MB)

0xc0ffffff  ┐
            │ Kernel区      0x01000000  ┐
0xc0000000  ┘──────┐                   │ 用户进程物理页
                   │                   │ (由mm_malloc分配)
                   └──────────────>    ┘
0x0804a000  ┐
            │ 用户进程      0x00ffffff
            │ (代码/数据)   0x00000000  ┐
0x08048000  ┘──────┐                   │ Kernel物理页
                   │                   │ (16MB)
0x00000000         └──────────────>    ┘
```

#### 3.“在Kernel完成页表初始化前，程序无法访问全局变量”这一表述是否正确？在init_page()里面我们对全局变量进行了怎样的处理？

不正确。在页表初始化前，程序使用物理地址访问全局变量。`init_page()`中通过建立了虚拟地址和物理地址的映射，使得页表初始化后Kernel仍能通过虚拟地址正常访问这些全局变量。

### 实验过程 （ai总结）

1. 修改`kernel/Makefile`：将`-Ttext=0x30000`改为`-Ttext=0xc0030000`
2. 修改`testcase/Makefile`：去掉`-Ttext`选项，使代码段从`0x8048000`开始
3. 在`include/config.h`中定义`IA32_PAGE`宏
4. 在`nemu/include/cpu/reg.h`的CPU_STATE中添加CR3寄存器
5. 实现`nemu/src/memory/mmu/page.c`中的`page_translate()`函数：根据线性地址的DIR、PAGE、OFFSET三部分查询二级页表，返回物理地址
6. 修改`nemu/src/memory/memory.c`中的`laddr_read()`和`laddr_write()`：当CR0的PG位为1时调用`page_translate()`，并处理跨页边界访问
7. 修改`kernel/src/elf/elf.c`中的`loader()`：使用`mm_malloc()`为用户进程的每个segment分配物理内存并建立页表映射
